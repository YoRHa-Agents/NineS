## 1. Background

NineS reached v3.0.0 with the [Knowledge Graph Analysis Engine](../changelog.md) (Understand-Anything decomposition + 24-dim self-eval). The user's feedback (`.local/feedbacks/feedback_for_v2.2.0.md`) framed the v2.2.0 work as four sequential asks:

1. **Deep self-iteration combined with EvoBench paradigm extension**, listing optimisable surface area.
2. **Verify against decomposition samples** so the evidence is concrete.
3. **Update the various reference repos** (NineS's `references/` folder) to absorb the new paradigm knowledge.
4. **Form an empirically-backed accept-list for the decision-maker** so v2.2.0 ships with verified data, not aspirations.

This document is the public-facing summary of how that work landed. It complements the internal artifacts under `.local/v2.2.0/` (survey, profile, design, benchmark, validate stages) and the four new `references/*.md` files that codify the patterns for downstream contributors.

The motivating observation: **NineS's v3.0.0 self-eval reported `overall = 0.9697` with 19 of 20 capability dims pinned at 1.000**, but the empirical profile (caveman / DevolaFlow / Understand-Anything as samples) confirmed the headline number was structurally untrustworthy. The work below decomposes that finding into 11 concrete gaps and 12 candidate fixes, six of which landed as POCs in this branch.

## 2. Survey Findings

### 2.1 Top 5 EvoBench gaps (vs NineS v3.0.0)

The full migration analysis lives in `.local/v2.2.0/survey/01_evobench_gap_analysis.md`. The five highest-leverage gaps:

1. **Cross-artifact consistency audit** (EvoBench v2.2.0 changelog: "deep cross-file data-consistency audit") — NineS emits 4-5 JSON files per analyze run with *no* cross-file integrity check.
2. **Typed `TaskInput` factory + project-aware `EvaluationContext`** — NineS evaluators take `src_dir: str = "src/nines"` defaults, so foreign-repo runs silently re-evaluate NineS itself.
3. **Weighted `MetricRegistry`** (EvoBench `ageneval-dimensions` `MetricDefinition`) — NineS uses unweighted means; saturation hides regressions.
4. **`with_retry` + `CostTracker` parallel runner** (EvoBench `ageneval-runner` `retry.rs` + `parallel.rs` slim) — NineS's `eval_max_retries` config knob is dead code; no per-evaluator wall-clock budget.
5. **DeterministicMockExecutor** (EvoBench `ageneval-cli`) — NineS CI runs live `pytest --collect-only` against itself; structural runner regressions hide behind real-environment noise.

### 2.2 Top 5 reference-repo paradigms

The full catalog lives in `.local/v2.2.0/survey/02_reference_repo_catalog.md` and covers 33 reference trees under `/home/agent/reference/`. The five most NineS-relevant cross-cutting paradigms:

1. **AgentBoard analytical evaluation breakdown** (`R1`) — fine-grained sub-skill panels + trajectory visualisation; the canonical fix for the §4.10 saturation problem.
2. **HAL holistic harness + trace governance** (`R3`) — unified `hal-eval` CLI across SWE-bench / USACO / AppWorld / τ-bench; encrypted trace upload pattern.
3. **τ-bench simulated user + Pass^k** (`R11`) — policy-grounded tool-dialog evaluation with consistency metrics across independent attempts.
4. **nearai-bench trajectory assertions + identity files** (`R16`) — per-turn assertions with `SOUL.md`/`IDENTITY.md` workspace identity files mirroring production prompts.
5. **ContextOS cognition primitives + GSD canonical gates** (`R7` + `R9`) — cognitive primitives between retrieve and generate; explicit `pre-flight / revision / escalation / abort` gate taxonomy.

These paradigms drove the reference-repo updates: four new `references/*.md` files in this release codify the patterns for downstream contributors (see §6).

## 5. POC Outcomes (S04)

Six POCs landed on `feat/v2.2.0-paradigm-extension` with **+72 net unit tests (1189 → 1261, +6.0 %), zero regressions**. Total LOC ≈ 2 280 across 9 new source files, 9 new test files, 6 modified source files. Each POC was independently re-validated in S05 (`.local/v2.2.0/validate/01_empirical_validation.md`).

| C02 | Project-scoped finding-IDs         | **CONFIRMED**  | 271 / 271 unique IDs (was 265 / 271); 3 distinct fingerprints (`45961152`, `05645815`, `8c1e11cc`)        |

### 5.2 Two PARTIAL verdicts in detail

**C04 (PARTIAL):** the runner-level wrap works correctly — caveman self-eval terminates in 33.3 s vs the >195 s baseline hang. But:
- `_build_json_output` in `src/nines/cli/commands/self_eval.py` (lines 189-217) does **not** include `report.timeouts` despite `SelfEvalReport.to_dict()` exposing it. Operators running `nines self-eval --format json` cannot detect partial runs (**N1 risk**).
- The four `Live*` evaluators that shell out (`LiveCodeCoverageEvaluator`, `LiveTestCountEvaluator`, `LintCleanlinessEvaluator`, `AgentAnalysisQualityEvaluator`) still have hard-coded subprocess timeouts (120 / 300 / 300 s); the design's POC item to wire `subprocess.run(timeout=min(dim_budget.hard_seconds, current_default))` was deferred (**N2 risk**).
- DevolaFlow + UA termination not re-validated; only caveman tested.

**C06 (PARTIAL):** the executor primitive ships with byte-identical determinism, distinctness, and seed isolation. But the design's actual value-prop was the **golden harness** — `MockEvaluator(DimensionEvaluator)`, fixtures at `tests/data/golden/self_eval_fixtures/*.json`, `tests/iteration/test_self_eval_golden.py` with the joint hang-detection test (a `MockEvaluator` doing `time.sleep(120)` should complete in ≤ 5 s after C04 wires budgets), and the silent-fallback regression detector. None of these were built. The shipped C06 is approximately **10 %** of the design's surface (**N4 risk** — blocks downstream C08 and C12).

### 5.3 Six new risks surfaced from S04 commits

These were not in the original design risk register; they emerged during implementation. Full table in `.local/v2.2.0/validate/03_consolidated_findings.md` §3.2.

- **N1: CLI JSON output drops new fields silently.** Future `context_fingerprint` (C01), `weighted_overall` (C08), `breakdown` (C12) will likely have the same problem unless `_build_json_output` is refactored to forward `report.to_dict()`.
- **N2: Daemon-thread cancellation can't kill `subprocess.run` hangs.** v2.3 child-process executor closes this fully; Wave 1 follow-up wires explicit subprocess timeouts.
- **N3: Verifier-side fix without builder-side fix is fragile.** New `KnowledgeGraph` consumers can re-introduce §4.1 mismatch.
- **N4: C06 shipped only the executor primitive, not the harness.** Block C08 + C12 on full-C06.
- **N5: Lint debt grew silently.** `ruff check src/nines` 6 → 15 errors (+9 stylistic); pre-commit hook recommended.
- **N6: `make typecheck` not run in S04.** Treat as merge-blocker for v2.2.0 release.

### 7.1 Per-reference summary

**`cross-artifact-audit.md`** (~241 lines, tier 2) — codifies the three-stage **verify → audit → gate** pattern. Per-artifact verification (`GraphVerifier`) confirms within-file integrity; cross-artifact audit (`ConsistencyAuditor`, planned C10) confirms multi-file invariants; release-blocking gates (`QualityGateMachine`, planned C07) wrap them in a state machine that refuses to write artifacts (or refuses CI exit code) when validation fails. Includes a five-method audit table mapping each `audit_*` to the §4 gap it covers, plus a developer workflow for adding new analyzer outputs.

**`resilience-budgets.md`** (~296 lines, tier 2) — codifies the three-primitive **`TimeBudget` + `RetryPolicy` + `CostBudget`** family. Documents the C04 daemon-thread design (with the `ThreadPoolExecutor` → `threading.Thread(daemon=True)` reasoning that emerged in S04), the C05 `with_retry` exponential-backoff helper, and the `CostBudget.add()` "raise after updating counters" contract that lets `EvalRunner.run` produce partial-results reports. Includes a "When to opt in" matrix mapping evaluator types to recommended budgets.

**`project-identity.md`** (~311 lines, tier 2) — codifies the **identity-as-explicit-input** principle. The C02 `project_fingerprint(project_root)` helper plus `format_finding_id` / `parse_finding_id` are documented at the API level; the planned C01 `EvaluationContext` is documented as the consumer. Recommends C01 *call* the C02 helper rather than reinventing — a refinement that emerged in S05's analytical validation.

**`derived-metrics.md`** (~342 lines, tier 2) — codifies the **algebra-over-constants** principle and the `formula_version` schema-migration pattern. Documents the C09 `economics` formula at line-level (with the per-sample empirical table from `c09_economics_variance.txt`) plus the planned C08 `MetricRegistry` with proposed per-dim thresholds. Pre-documents the +0.05 `economics_score` lift that will land "for free" when C11a adds 4 new ContextOS-derived mechanism categories — so reviewers don't read it as a regression.

## 8. Final Recommendation

Ship **Wave 1 (v2.2.0) only after the 6.5-day follow-up work lands**, particularly the C04 CLI-JSON `timeouts` exposure and the C03 builder-side fix. Without those two, the v2.2.0 release will reach end-users with a measurable but invisible improvement (C04) and a fragile fix that re-breaks if any new graph-consumer is added (C03). The 6 POCs themselves are sound; the gap is between "code that works in the lab" and "code that ships its value to operators".

After v2.2.0, run **Wave 2 as a single sprint focused on C01 first** (a two-week effort that unlocks every downstream candidate's empirical promise). Block Waves 3 and 4 on Wave 2's completion — there is no point shipping weighted scoring (C08) or mechanism diversification (C11) when the underlying inputs still collapse to NineS's own values.

The empirical case for v2.2.0 is real — but the deliverable that goes to users in v2.2.0 should be **the closure of §4.1 / §4.2 / §4.5 / §4.6 plus a bounded §4.7**, not "all of paradigm-extension". The differentiation track's headline score-shape changes belong in v2.3 with proper documentation and a clean dual-emit migration.

## 10. Operator Implications

For NineS operators (running `nines analyze` / `nines self-eval` / `nines benchmark` against target repos), the v2.2.0-rc1 release means:

1. **No more silent finding-ID collisions.** Every report now carries an 8-char `project_fingerprint` prefix on `AI-` IDs (see `c02_id_uniqueness.txt`). Cross-run dashboards can dedupe on `(project_id, prefix, idx)` tuples instead of raw IDs.
2. **No more 5-minute self-eval hangs.** Pass `--evaluator-timeout 60` (default) to `nines self-eval`; runaway evaluators are bounded; `report.timeouts` records which dim breached. **N1 caveat:** the field is currently dropped from `--format json` payload — operators relying on JSON should wait for the Wave 1 follow-up.
3. **No more `verification.passed = False` reports.** All graph-strategy analyses on caveman / DevolaFlow / UA now pass the verifier. Wave 1 follow-up adds `--strict-graph` so future verification failures will exit non-zero.
4. **`break_even_interactions` is now a real number.** Previously constant `2`, now `ceil(overhead / saved_per_interaction)`. The `formula_version=2` field signals the new shape; legacy parsers continue to read `formula_version=1` reports.
5. **No project-context magic yet.** Foreign-repo self-eval still falls back to NineS's own values when `--src-dir` is omitted (§4.8 gap). Wave 2 (C01) ships `EvaluationContext` to fix this; until then, always pass `--src-dir` and `--test-dir` explicitly when running against non-NineS targets.

For NineS contributors (writing new analyzers, evaluators, or iteration logic), the v2.2.0 reference docs are the canonical entry points — load them by trigger word from the agent's `references/index.md` v1.1.0 catalog.
