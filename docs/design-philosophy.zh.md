# 设计理念

NineS 基于一系列深思熟虑的设计决策构建，这些决策塑造了每个模块、协议和工作流。本页阐述了这些决策背后的思考以及驱动项目前进的愿景。

---

## 愿景

YoRHa-Agents 团队致力于构建自我改进的 AI Agent 基础设施。我们的目标是创建能够自我评估、从环境中学习并迭代改进的系统——形成自省与成长的递归循环。

NineS 通过三个集成的能力顶点体现这一愿景：**评估（Evaluation）**、**采集（Collection）** 和 **分析（Analysis）**。结合 **MAPIM 自迭代循环**，这些顶点形成一个闭环系统，其中度量驱动改进，改进驱动新的度量。

我们相信，AI Agent 开发的未来不在于静态基准测试，而在于与被评估的 Agent 共同演化的活跃评估系统。

---

## 核心原则

### 三顶点协同（Three-Vertex Synergy）

评估、采集和分析不是孤立的子系统——它们构成一个**自我强化的循环**。每个顶点向其他顶点输送数据，创造涌现智能：

- **评估缺口**触发定向信息采集（V1 → V2）
- **采集到的仓库**成为分析目标（V2 → V3）
- **分析产出的知识单元**成为评估任务候选（V3 → V1）
- **知识缺口**生成新的搜索查询（V3 → V2）
- **采集的数据**生成评估基准（V2 → V1）
- **评估分数**输入 MAPIM 自改进循环（V1 → Iteration）

这种六流架构意味着改进任何单个顶点都会在整个系统中产生级联收益。整体大于部分之和。

### 自迭代（MAPIM）

NineS 通过结构化自省进行自我改进，使用 **MAPIM 循环**：

1. **Measure（度量）** — 评估全部 19 个维度以建立当前性能基线
2. **Analyze（分析）** — 检测当前分数与目标之间的差距；按严重程度排序
3. **Plan（计划）** — 生成具体的模块级改进行动（每次迭代 ≤3 个）
4. **Improve（改进）** — 对代码库执行计划的改进行动
5. **Measure（度量）** — 重新评估以量化变更的影响

此循环受 **kaizen（持续改善）** 和**控制论反馈环**的启发。系统不追求一次性达到完美——而是通过多次经过验证的小步骤收敛。4 方法收敛检测器确保循环在达到真正的稳定性时终止，既不会过早也不会过晚。

### 协议驱动的可扩展性（Protocol-Driven Extensibility）

NineS 使用 Python `Protocol` 类（PEP 544）实现**结构化子类型**。任何匹配所需方法签名的类都满足协议——无需继承或注册仪式。

这意味着：

- 第三方 scorer、collector 和 analyzer 无需了解 NineS 基类即可工作
- 扩展代码与 NineS 内部实现零耦合
- 测试简单——任何匹配协议签名的 mock 对象都是有效的
- 系统可以演进其协议而不破坏现有扩展

关键协议包括 `Scorer`、`SourceProtocol`、`DimensionEvaluator`、`PipelineStage` 和 `SkillAdapterProtocol`。

### 确定性可复现（Deterministic Reproducibility）

NineS 中的每次评估运行都可以**精确复现**。主种子通过所有随机操作传播：

- 任务选择和排序
- 沙箱环境创建
- Scorer 随机化（用于模糊匹配）
- 矩阵采样策略

通过固定种子（`--seed 42`），在相同代码库上的两次运行将产生相同的结果。这对于科学严谨性、调试和基线比较至关重要。

### 默认隔离（Isolation by Default）

NineS 对所有评估执行强制实施**三层沙箱**：

1. **进程隔离** — 使用 `RLIMIT_AS` 和进程组控制的子进程执行
2. **虚拟环境** — 通过 `uv` 或标准库为每个沙箱创建全新的 venv，防止依赖污染
3. **文件系统隔离** — 临时目录作用域，带有执行前后的快照差异比较

这种方法**无需 Docker** 即可提供强大的隔离，使 NineS 可以在任何安装了 Python 3.12+ 的机器上运行。`PollutionDetector` 在每次执行后跨 4 个维度（环境变量、文件、目录、`sys.path`）验证宿主机完整性。

### 零配置启动（Zero-Configuration Start）

NineS 遵循**渐进式配置深度**模型：

1. **CLI 标志** — 覆盖一切，用于快速一次性运行
2. **项目配置** — 项目根目录中的 `nines.toml`，用于团队共享设置
3. **用户配置** — `~/.config/nines/config.toml`，用于个人偏好
4. **内置默认值** — `src/nines/core/defaults.toml`，提供开箱即用的合理行为

新用户可以立即运行 `nines eval tasks/coding.toml`。随着需求增长，可以添加配置层而不干扰现有工作流。

## 架构决策

### 为什么选择 TOML 而非 YAML/JSON

| 因素 | TOML | YAML | JSON |
|------|------|------|------|
| 可读性 | 优秀 | 良好（缩进敏感） | 较差（语法冗长） |
| 类型安全 | 原生支持（int、float、datetime、bool） | 隐式转换（1.0 vs "1.0" 歧义） | 有限（无 datetime、无注释） |
| Python 生态 | 标准库 `tomllib`（3.11+） | 需要 `pyyaml` | 内置 `json` |
| 注释支持 | 是 | 是 | 否 |
| 嵌套深度 | 天然浅层 | 无限制（可能难以阅读） | 无限制 |

TOML 是 Python 生态系统中项目元数据的标准格式（`pyproject.toml`）。将其用于 NineS 配置保持了一致性，并避免了 YAML 众所周知的问题（隐式类型转换、挪威问题等）。

### 为什么选择 SQLite 而非 PostgreSQL

NineS 被设计为**单用户、本地优先的工具**。SQLite 提供：

- 零配置——无服务器进程、无凭证、无网络
- 单文件存储——易于备份、版本管理和分发
- WAL 模式——并发读取与写入串行化
- FTS5——内置全文搜索，用于知识索引
- 足够的性能——NineS 的工作负载完全在 SQLite 的能力范围内

PostgreSQL 会增加运维复杂度而不带来实质性收益。存储层通过协议抽象，如果出现多用户场景，未来的迁移将很简单。

### 为什么选择 Protocol 而非抽象基类

| 方面 | Protocol | ABC |
|------|----------|-----|
| 耦合度 | 零——结构化子类型 | 需要继承 |
| 第三方兼容性 | 与任何匹配的类兼容 | 必须继承或注册 |
| 运行时检查 | `@runtime_checkable` 可选 | `isinstance()` 始终可用 |
| 测试 | 任何匹配的 mock 都有效 | 必须继承或使用 `ABC.register()` |
| 组合 | 天然可组合 | 菱形继承问题 |

Protocol 与 NineS 的可扩展性目标一致：第三方代码应该通过匹配签名来工作，而不是通过继承一棵树。

### 为什么选择 Token-Bucket 速率限制

外部 API 源（GitHub、arXiv）有不同的速率限制。Token-bucket 提供：

- **按源校准** — 每个 collector 获得与 API 限制匹配的独立令牌桶
- **突发容忍** — 允许短时间突发而不触及速率限制
- **自适应退避** — 收到 HTTP 429 时，令牌桶排空并缓慢补充
- **可预测吞吐量** — 稳态采集速率是明确定义的

对于 NineS 的采集模式，这比固定窗口或滑动窗口方案更简单且更可预测。

### 为什么使用 4 方法收敛检测

单一收敛方法可能被噪声或瞬态平台期欺骗。NineS 在四种统计方法之间使用**多数投票**：

1. **Sliding Window Variance** — 近期分数的低方差表明稳定性
2. **Relative Improvement** — 递减的收益表明增益在减小
3. **Mann-Kendall Trend Test** — 无统计显著趋势（95% 置信度）
4. **CUSUM** — 累积和控制图检测均值偏移

当 **≥3/4 的方法一致** 时宣布收敛。这种复合方法在保持对单个方法失败的鲁棒性的同时，提供了统计严谨性。

## NieR 的联结

NineS 的命名和主题灵感来自**尼尔：机械纪元（NieR: Automata）**，这款动作 RPG 探索了机器意识、自我认知以及毁灭与创造循环的主题。

**9S（YoRHa No.9 Type S）** 是扫描型（Scanner-type）机器人——善于分析、充满好奇、致力于理解周围系统。这使 9S 成为一个分析和评估系统的完美命名来源，该系统检查、度量并寻求理解 AI Agent 的能力。

更广泛的 **YoRHa-Agents** 项目取材于游戏的核心主题：

- **机器质疑自身的目的** — NineS 通过 19 个自评估维度持续评估自身组件是否在履行其预期角色
- **通过递归产生自我意识** — MAPIM 循环创造了递归式自我改进，映射了游戏中关于自我引用产生意识的探索
- **毁灭与创造的循环** — 每次迭代都可能摧毁旧的假设（差距检测）并创造新的能力（改进规划）
- **努力的徒劳与美丽** — 即使收敛被逼近但永远无法完美达到，每次迭代都使系统更接近其理想

命名惯例贯穿整个项目：系统以分析型单位命名，团队以部署这些单位的机器人特遣队命名。

---

## 路线图愿景

NineS 正在朝着更深层次的自我意识和更广泛的能力方向发展：

- **更多评估维度** — 超越当前的 19 个维度，新维度将随着领域的发展捕捉新兴的 Agent 能力
- **更丰富的采集源** — HuggingFace、PyPI、会议论文集和社交媒体跟踪将充实知识图谱
- **更深入的分析能力** — LLM 增强的代码审查、语义搜索和多语言 AST 分析（通过 tree-sitter 支持 TypeScript、Go、Rust）
- **更紧密的自改进循环** — 自动课程生成将创建针对检测到的弱点的评估任务，CI/CD 集成将使自评估成为合并门控
- **完整的自动课程** — 系统最终将基于能力差距生成自己的训练练习，闭合评估与改进之间的循环

长期愿景是一个不仅能评估 AI Agent，还能根据 AI 能力变化的格局演进其评估标准的系统——一个与领域共同成长的活跃基准。
