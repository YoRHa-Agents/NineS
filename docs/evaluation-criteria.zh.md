# 评估标准

NineS 使用 19 个可度量的维度，分为 4 个类别，来评估和改进 AI Agent 的能力。本页详细介绍每个维度、评分方法、收敛检测以及如何在实践中使用这些标准。

---

## 概述

自评估是 NineS 自我改进能力的基础。NineS 不依赖外部审查者，而是在全面的维度集合上度量自身性能——从评分准确性和数据完整性到流水线延迟和跨顶点协同。

每个维度具有：

- 定义了数据来源的**具体度量方法**
- 产生归一化分数（0–100）的**公式**
- **改进方向**（越高越好或越低越好）
- 定义最低可接受性能的 **MVP 目标值**

19 个维度涵盖所有三个能力顶点以及系统级健康状况，确保自评估捕捉 NineS 运营状态的完整画面。

---

## 评估方法论

### 综合评分

NineS 使用可配置的权重将各维度分数聚合为单一综合分数：

```
V1_score = weighted_mean(normalized(D01..D05))
V2_score = weighted_mean(normalized(D06..D10))
V3_score = weighted_mean(normalized(D11..D15))
system_score = weighted_mean(normalized(D16..D19))

composite = 0.30 × V1 + 0.25 × V2 + 0.25 × V3 + 0.20 × system
```

越低越好的维度（D07、D16）在聚合前通过 `1 − min(value, cap) / cap` 进行反转。D19（协同）被裁剪到 [0, 1]。权重可通过 `nines.toml` 中的 `[self_eval.weights]` 配置。

### 统计可靠性

NineS 采用多种统计度量来确保评估结果可信：

- **pass@k** — k 次尝试中至少一次通过的概率。估计能力广度。
- **Pass^k** — k 次尝试都产生相同结果的概率。度量确定性。
- **Pass³** — 三次运行一致性检查，用于基线稳定性验证。
- **Bootstrap 置信区间** — 分数估计的非参数置信界限。

### 矩阵评估

`MatrixEvaluator` 支持在 N 个可配置轴上进行评估（任务难度、scorer 类型、超时时间等）。这实现了：

- 评估配置的组合覆盖
- 控制总评估成本的预算守卫
- 大参数空间的采样策略
- 按单元格和跨轴分析

### 基线比较与差距检测

基线在某个时间点冻结维度分数。`GapDetector` 将当前分数与基线进行比较，以：

- 识别**回退** — 变差的维度
- 识别**改进** — 变好的维度
- 将差距分类为 **critical**、**major**、**minor** 或 **acceptable**
- 提供指向特定模块的**根因提示**

## 维度类别

### V1 Evaluation（D01–D05）

这些维度评估评估流水线本身的健康性和准确性。

#### D01: Scoring Accuracy

| 字段 | 值 |
|------|---|
| **度量内容** | NineS 的自动化评分与人工验证的真实值的接近程度 |
| **度量方法** | 在 ≥30 个黄金测试任务上运行 `EvalRunner`。将输出分数与真实标签进行比较。 |
| **公式** | `accuracy = count(|nines_score − golden_score| ≤ 0.05) / total_tasks` |
| **数据来源** | `data/golden_test_set/`，包含 `expected_score` 字段 |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.90 |
| **重要性** | 如果 scorer 不准确，所有下游决策（差距检测、改进规划）都建立在虚假数据上。评分准确性是评估流水线的基石。 |
| **改进方向** | 扩展黄金测试集，重新校准 scorer 阈值 |

#### D02: Evaluation Coverage

| 字段 | 值 |
|------|---|
| **度量内容** | NineS 能成功加载、执行和评分的已定义任务类型的比例 |
| **度量方法** | 枚举 schema 中的所有任务类型。对每种类型，尝试 load → execute → score。 |
| **公式** | `coverage = successful_types / total_defined_types` |
| **方向** | 越高越好 |
| **MVP 目标** | 1.00 |
| **重要性** | 任务类型支持的缺口意味着评估的盲点。完全覆盖确保没有能力未被度量。 |
| **改进方向** | 实现缺失的任务类型加载器 |

#### D03: Reliability (Pass^k Consistency)

| 字段 | 值 |
|------|---|
| **度量内容** | 相同任务在独立运行中是否产生相同的通过/失败结果 |
| **度量方法** | 运行 ≥20 个任务 × k=3 次独立运行，在全新沙箱中使用相同种子。 |
| **公式** | `consistency = count(all_k_agree) / total_tasks` |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.95 |
| **重要性** | 不确定的评估结果会破坏信任。如果相同输入产生不同输出，分数对于趋势分析和收敛检测是不可靠的。 |
| **改进方向** | 修复沙箱中的非确定性，收紧种子控制 |

#### D04: Report Quality

| 字段 | 值 |
|------|---|
| **度量内容** | 生成的报告是否包含所有必需章节且内容有效 |
| **度量方法** | 解析 `MarkdownReporter` 和 `JSONReporter` 输出。检查必需章节。 |
| **公式** | `quality = present_valid_sections / total_required_sections` |
| **必需章节** | summary、task_results、per_dimension_scores、statistical_summary、recommendations、metadata |
| **方向** | 越高越好 |
| **MVP 目标** | 1.00 |
| **重要性** | 报告是 NineS 与用户之间的主要接口。缺失的章节使结果更难以解读和采取行动。 |

#### D05: Scorer Agreement

| 字段 | 值 |
|------|---|
| **度量内容** | 不同 scorer 实现对相同输出的一致程度 |
| **度量方法** | 使用所有适用的 scorer 评分 ≥20 个输出。计算成对 Cohen's κ。 |
| **公式** | 所有 scorer 对的平均成对 κ |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.70 |
| **重要性** | 如果 scorer 之间存在显著分歧，scorer 的选择就成为主导变量——而非 Agent 输出的质量。一致性验证了 scorer 的可靠性。 |
| **改进方向** | 重新校准 scorer 参数，添加校准数据集 |

### V2 Search（D06–D10）

这些维度评估信息采集的质量和效率。

#### D06: Source Coverage

| 字段 | 值 |
|------|---|
| **度量内容** | 已配置的数据源中处于活跃和可达状态的比例 |
| **度量方法** | 对每个已配置的源执行轻量级健康检查查询。 |
| **公式** | `coverage = active_sources / configured_sources` |
| **方向** | 越高越好 |
| **MVP 目标** | 1.00 |
| **重要性** | 不活跃的源在数据采集中造成盲点。完全的源覆盖确保全面的信息收集。 |
| **改进方向** | 修复 API 凭证，检查网络连接 |

#### D07: Tracking Freshness

| 字段 | 值 |
|------|---|
| **度量内容** | NineS 检测被跟踪实体变更的速度 |
| **度量方法** | 跟踪具有已知变更计划的金丝雀实体。度量检测延迟。 |
| **公式** | `median(detection_time − change_time)`，跨所有金丝雀 |
| **归一化** | 反转：`1 − min(lag, 120) / 120` |
| **方向** | 延迟越低越好（归一化分数越高越好） |
| **MVP 目标** | ≤60 min |
| **重要性** | 过时的数据导致过时的分析。新鲜的跟踪确保 NineS 基于最新信息运作。 |

#### D08: Change Detection Recall

| 字段 | 值 |
|------|---|
| **度量内容** | NineS 成功检测到的实际变更的比例 |
| **度量方法** | 将检测到的变更与 7 天窗口内的真实变更日志进行比较。 |
| **公式** | `recall = true_positives / (true_positives + false_negatives)` |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.85 |
| **重要性** | 遗漏的变更意味着遗漏的信息。高召回率确保重要更新不会被忽视。 |
| **改进方向** | 修复分页，扩大查询范围 |

#### D09: Data Completeness

| 字段 | 值 |
|------|---|
| **度量内容** | 采集到的实体中字段的填充程度 |
| **度量方法** | 查询所有实体。按实体模型 schema 检查已填充字段。 |
| **公式** | `mean(populated_fields / total_fields)`，跨所有实体 |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.90 |
| **重要性** | 部分填充的记录降低了下游分析的价值。完整的数据能够产生更丰富的洞察。 |

#### D10: Collection Throughput

| 字段 | 值 |
|------|---|
| **度量内容** | NineS 从外部源采集实体的速率 |
| **度量方法** | 记录每个源采集的实体数量和实际耗时。 |
| **公式** | `entities / duration_minutes` |
| **方向** | 越高越好 |
| **MVP 目标** | ≥50 |
| **重要性** | 缓慢的采集会成为整个流水线的瓶颈。足够的吞吐量确保数据的及时可用性。 |

### V3 Analysis（D11–D15）

这些维度评估代码分析和知识抽取的质量。

#### D11: Decomposition Coverage

| 字段 | 值 |
|------|---|
| **度量内容** | 分解器捕获的可分析代码元素的比例 |
| **度量方法** | 运行 AST 提取器计数所有元素。运行 Decomposer。比较。 |
| **公式** | `captured_elements / total_analyzable_elements` |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.85 |
| **重要性** | 遗漏的元素就是遗漏的知识。高覆盖率确保对代码库的全面理解。 |

#### D12: Abstraction Quality

| 字段 | 值 |
|------|---|
| **度量内容** | NineS 识别代码中设计模式的准确程度 |
| **度量方法** | 在人工标注的参考代码库上运行模式检测。 |
| **公式** | 跨架构标签的宏平均 F1 |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.60 |
| **重要性** | 正确的模式识别使有意义的架构分析成为可能。错误分类导致对结构的不正确理解。 |

#### D13: Code Review Accuracy

| 字段 | 值 |
|------|---|
| **度量内容** | 自动化代码审查发现的准确性 |
| **度量方法** | 在具有已知问题的标注代码文件上运行 `CodeReviewer`。 |
| **公式** | F1 = 2 × (precision × recall) / (precision + recall) |
| **匹配标准** | 正确的文件、行范围（±3 行）和问题类别 |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.70 |
| **重要性** | 不准确的代码审查会在误报上浪费开发者注意力，或遗漏真正的问题。精确率和召回率都必须高。 |

#### D14: Index Recall

| 字段 | 值 |
|------|---|
| **度量内容** | 知识索引为查询检索相关结果的效果 |
| **度量方法** | 对索引的知识单元执行 ≥15 个基准查询。 |
| **公式** | `mean(relevant_in_top_10 / total_relevant)` |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.70 |
| **重要性** | 遗漏相关结果的知识索引违背了其目的。高召回率确保搜索返回用户需要的内容。 |

#### D15: Structure Recognition

| 字段 | 值 |
|------|---|
| **度量内容** | 跨代码库的架构模式识别的准确性 |
| **度量方法** | 在 5 个具有已知架构的参考代码库上运行 `StructureAnalyzer`。 |
| **公式** | `correctly_identified / total_annotated_patterns` |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.60 |
| **重要性** | 结构理解是分解和分析建议的基础。错误识别会在所有下游输出中产生级联效应。 |

### System-Wide（D16–D19）

这些维度评估系统整体健康状况和横切关注点。

#### D16: Pipeline Latency

| 字段 | 值 |
|------|---|
| **度量内容** | 评估流水线的端到端执行时间 |
| **度量方法** | 对 `EvalRunner` 进行计时检测。运行黄金测试集。 |
| **公式** | 所有任务的 p50 实际耗时 |
| **归一化** | `1 − min(p50, 300) / 300` |
| **方向** | 延迟越低越好（归一化分数越高越好） |
| **MVP 目标** | ≤30s |
| **重要性** | 缓慢的流水线降低迭代速度。MAPIM 循环的有效性取决于快速反馈。 |

#### D17: Sandbox Isolation

| 字段 | 值 |
|------|---|
| **度量内容** | 沙箱化执行是否使宿主机环境保持清洁 |
| **度量方法** | 使用 `execute_with_pollution_check()` 包装每次执行。 |
| **公式** | `clean_runs / total_runs` |
| **方向** | 越高越好 |
| **MVP 目标** | 1.00 |
| **重要性** | 任何沙箱污染都是严重 bug。宿主机污染可能破坏结果、影响后续运行并损害对隔离层的信任。 |
| **严重性** | 任何低于 1.0 的值都会触发立即调查 |

#### D18: Convergence Rate

| 字段 | 值 |
|------|---|
| **度量内容** | MAPIM 循环达到稳定性的效率 |
| **度量方法** | 运行 MAPIM 循环。记录达到 4 方法收敛的迭代次数。 |
| **公式** | `1 − (iterations_to_converge / max_iterations)` |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.50 |
| **重要性** | 需要太多迭代才能收敛的系统浪费计算资源和时间。高效的收敛意味着改进行动是有效的。 |

#### D19: Cross-Vertex Synergy

| 字段 | 值 |
|------|---|
| **度量内容** | 一个顶点的改进是否导致其他顶点的改进 |
| **度量方法** | 提取每次迭代的逐顶点分数。计算滞后相关性。 |
| **公式** | `mean(corr(ΔVa[t], ΔVb[t+1]))`，跨所有 6 个有向顶点对 |
| **最小数据量** | 需要 ≥5 个迭代数据点 |
| **方向** | 越高越好 |
| **MVP 目标** | ≥0.00 |
| **重要性** | 正协同意味着三顶点架构按设计工作——改进会级联。零或负协同表明各顶点在各自为政。 |

## 评分系统

### 逐维度评分

每个维度产生一个原始值，归一化到 0–100 的范围：

- **越高越好**的维度：分数为原始值缩放到 [0, 100]
- **越低越好**的维度（D07、D16）：通过 `1 − min(value, cap) / cap` 反转后缩放
- **有界维度**（D19）：裁剪到 [0, 1] 后缩放

### 类别聚合

维度按四个类别分组，使用加权平均：

| 类别 | 维度 | 默认权重 |
|------|------|---------|
| V1 Evaluation | D01–D05 | 0.30 |
| V2 Search | D06–D10 | 0.25 |
| V3 Analysis | D11–D15 | 0.25 |
| System-wide | D16–D19 | 0.20 |

### 综合分数

综合分数是类别分数的加权平均。所有权重可通过 `nines.toml` 中的 `[self_eval.weights]` 配置。

### 阈值定义

| 级别 | 综合分数 | 含义 |
|------|---------|------|
| **Pass（通过）** | ≥0.85 | 系统在所有类别中表现良好 |
| **Warning（警告）** | 0.70–0.84 | 某些维度需要关注；MAPIM 应针对差距进行定向改进 |
| **Fail（失败）** | <0.70 | 检测到重大问题；需要立即采取行动 |

每个维度的阈值由其 MVP 目标值定义。低于目标的维度被标记为优先改进对象。

## 收敛检测

### 4 方法多数投票

当 **≥3/4** 的统计方法一致认为综合分数已稳定时，宣布收敛：

| 方法 | 收敛条件 | 默认参数 |
|------|---------|---------|
| Sliding Window Variance | σ²(最近 w 个分数) < τ | w=5, τ=0.001 |
| Relative Improvement | 平均步进改进 < threshold | threshold=0.005 |
| Mann-Kendall Trend Test | \|Z\| ≤ 1.96（无显著趋势） | 95% 置信度 |
| CUSUM Stability | 未检测到偏离参考均值的偏移 | h=1.0, δ=0.5 |

### 为什么使用多种方法

没有单一的统计检验能完美捕捉"系统已停止改进"：

- **滑动方差**可能被震荡但稳定的分数欺骗
- **相对改进**会遗漏分数先平台期后跳跃的情况
- **Mann-Kendall** 需要足够的数据点才有统计效力
- **CUSUM** 对参数选择敏感

通过要求 ≥3/4 一致，复合决策对任何单一方法的失败模式都具有鲁棒性。这避免了过早收敛（过早停止）和遗漏收敛（浪费迭代）。

## 使用评估标准

### 运行自评估

```bash
# 对全部 19 个维度进行完整自评估
nines self-eval

# 评估特定维度
nines self-eval --dimensions D01,D02,D03

# 生成详细报告
nines self-eval --report -o self_eval_report.md
```

### 与基线比较

```bash
# 将当前分数与存储的基线进行比较
nines self-eval --baseline v1 --compare

# 建立新基线
nines self-eval --baseline v1.1 --save
```

### 解读结果

1. **检查综合分数** — 系统是通过、警告还是失败？
2. **审查各类别分数** — 哪个顶点最需要关注？
3. **检查各维度** — 哪些具体维度低于目标？
4. **阅读差距分析** — 表现不佳的根因是什么？
5. **查看建议** — NineS 提出了哪些具体行动？

### 针对特定维度改进

当某个维度低于目标时，`GapDetector` 提供：

- **严重性分类** — critical、major、minor 或 acceptable
- **根因提示** — 哪个模块或组件是责任方
- **改进行动** — 提高分数的具体步骤

`ImprovementPlanner` 将这些差距映射到每次 MAPIM 迭代 ≤3 个模块级行动，确保聚焦且可管理的改进循环。
